# Territory IA MCP server (territory-ia)

> **Trust boundary.** Canonical tool schemas live in `[tools/mcp-ia-server/src/index.ts](../tools/mcp-ia-server/src/index.ts)` + the MCP server's `list_*` responses. This catalog is a human-readable overview and **can lag**. When a tool signature / allowed `caller_agent` / retired name appears here that contradicts the server, trust the server. Agents: fetch schemas via MCP, not this doc.

Recommended [Model Context Protocol](https://modelcontextprotocol.io/) server that exposes the **same on-disk information architecture** the repo already uses for agents: `ia/specs/*.md`, `ia/rules/*.md`, `glossary.md`, root docs such as `AGENTS.md` and `ARCHITECTURE.md` (via `buildRegistry()`), and `**BACKLOG.md`** (via the `backlog_issue` tool only—not listed in `list_specs`).

## Relationship to agent routing

Task-to-spec priorities match `**ia/rules/agent-router.md**`. That rule file also contains **“MCP — territory-ia”** with the default tool order when the server is enabled. The MCP does not replace rules or specs; it returns **slices** (sections, glossary rows, router table matches) so agents avoid loading multi-hundred-line files whole.

## Policy for Cursor agents

- **Terminology:** Tool names (`snake_case`) and descriptions should align with `[AGENTS.md](../AGENTS.md)` — glossary-backed domain terms, same vocabulary as specs and backlog. When adding or renaming tools, update this file and `[tools/mcp-ia-server/README.md](../tools/mcp-ia-server/README.md)` together with `registerTool` in code.
- **Glossary tools (`glossary_discover`, `glossary_lookup`):** The on-disk glossary is **English**. Agents must pass **English** in `query` / `keywords` / `term`. If the human conversation is in another language, **translate** the concepts into English (canonical domain words such as **street**, **road stroke**, **wet run**) before calling these tools. The server does not provide multilingual matching.
- **Workspace expectation:** This repo is set up so **Cursor** can run **territory-ia** from `.mcp.json` (optional `**REPO_ROOT`**; often `"."` relative to the workspace). Agents with tool access should **prefer MCP** for IA lookups in **Agent** chats. When `**REPO_ROOT`** is unset, the server walks up from `**process.cwd()**` for `config/postgres-dev.json` or `ia/specs/glossary.md`, so `**npm**` / `**tsx**` from `**tools/mcp-ia-server/**` still resolves the repo root. For `**project_spec_journal_***` and `**unity_bridge_***`, set `**DATABASE_URL**` in the MCP host environment if you need to override committed `[config/postgres-dev.json](../config/postgres-dev.json)` (read when `**DATABASE_URL**` is unset and not in **CI**). Otherwise tools use that file or return `**db_unconfigured`** when no URL resolves.
- **Human / IDE settings:** Whether tool runs require a click to approve is controlled by **Cursor** (e.g. auto-run or approval settings for MCP/tools)—not by this repo. Adjust in Cursor **Settings** if you want fewer prompts.
- **Not guaranteed every turn:** The model still chooses whether to call a tool; repo rules and `AGENTS.md` exist to **bias** behavior toward MCP first.
- **Cursor User Rules (optional):** In **Cursor Settings → Rules for AI** (or your global user rules), add a one-liner such as: *In the territory-developer workspace, prefer territory-ia MCP tools (`backlog_issue`, then spec/glossary/router tools) before reading whole spec files.* The repo cannot enforce IDE settings; this duplicates the intent of `AGENTS.md` for every chat.

## Issue kickoff workflow

When starting work on `**BUG-XX` / `FEAT-XX` / `TECH-XX`** (etc.), prefer `**issue_context_bundle({ issue_id })**` (pending registration — Stage 9+) for a one-shot composite load of issue yaml + spec + stage block + glossary anchors + invariants. Until registered, fall back to the sequential recipe: call `**backlog_issue**` with `issue_id` first, then `router_for_task` / `glossary_discover` / `glossary_lookup` / `spec_section` (or `**spec_sections**` when several slices are needed in one turn) as needed. `**status**` is `**open**` for rows in `**BACKLOG.md**` or `**completed**` when the checklist row is `**[x]**` in `**BACKLOG-ARCHIVE.md**`.

`**depends_on_status` / `soft_only`:** The parser treats an id as **soft** when it appears after the first `soft` token on the **Depends on** line (e.g. `Soft: **TECH-15`**) or when the line uses `****TECH-82** (soft: …)**` and the parenthetical does **not** cite a different issue id (so `**TECH-37** (soft: **TECH-38** …)` still leaves **TECH-37** as non-soft). Open soft dependencies get `**satisfied`: true** so `**plan-author`** (Stage 1×N) does not block on them (absorbs the retired `project-spec-kickoff` non-blocking semantics).

## Project spec workflows (Cursor Skills)

Repo **Cursor Skills** define **ordered** MCP usage for `ia/projects/{ISSUE_ID}.md`:

- **Create new issue + project spec stub from a prompt:** `[ia/skills/project-new/SKILL.md](../ia/skills/project-new/SKILL.md)` (**project-new** — trace in `[BACKLOG-ARCHIVE.md](../BACKLOG-ARCHIVE.md)`).
- **Review / enrich before code (Stage 1×N bulk):** `[ia/skills/plan-author/SKILL.md](../ia/skills/plan-author/SKILL.md)` — absorbs the retired `project-spec-kickoff` / `spec-kickoff` path per M6 collapse.
- **Execute Implementation Plan:** `[ia/skills/project-spec-implement/SKILL.md](../ia/skills/project-spec-implement/SKILL.md)`.
- **Close after verified work:** `[ia/skills/project-spec-close/SKILL.md](../ia/skills/project-spec-close/SKILL.md)` — persist IA (**glossary**, **reference specs**, `**ARCHITECTURE.md`**, rules, `**docs/**`), optional `**project_spec_journal_persist**` when `**DATABASE_URL**` is set delete `ia/projects/{ISSUE_ID}.md`, `npm run validate:dead-project-specs`, **remove** the row from `[BACKLOG.md](../BACKLOG.md)`, append `**[x]`** to `[BACKLOG-ARCHIVE.md](../BACKLOG-ARCHIVE.md)`, **purge** the closed id from durable docs and code (user-confirmed). **Closeout helpers:** MCP `**project_spec_closeout_digest`**, `**project_spec_journal_persist**`, `**spec_sections**`, and root `**npm run closeout:worksheet**`, `**closeout:dependents**`, `**closeout:verify**`, `**npm run db:persist-project-journal**`.
- **Post-implementation validation:** `[ia/skills/project-implementation-validation/SKILL.md](../ia/skills/project-implementation-validation/SKILL.md)` — optional ordered `**npm`** checks aligned with `[.github/workflows/ia-tools.yml](../.github/workflows/ia-tools.yml)`. From repo root, `**npm run validate:all**` runs **dead project spec** paths, `**npm run compute-lib:build`** (`**territory-compute-lib**`), `**test:ia**`, `**validate:fixtures**`, `**generate:ia-indexes --check**`; run `**npm ci**` under `**tools/compute-lib**` / `**tools/mcp-ia-server**` first if `**compute-lib:build**` or `**test:ia**` fails. Use after **MCP** / **schema** / index-source edits; **project-spec-close** may reference this before the mandatory `**validate:dead-project-specs`** cascade step.
- **Full local closed loop (canonical):** Root `**npm run verify:local`** (`[tools/scripts/verify-local.sh](../tools/scripts/verify-local.sh)`) runs `**validate:all**` then `[tools/scripts/post-implementation-verify.sh](../tools/scripts/post-implementation-verify.sh)` with `**--skip-node-checks**`: if `**Temp/UnityLockfile**` exists (**macOS**), **AppleScript** save/quit and wait up to **30s** → `**unity:compile-check`** → `**db:migrate**` → `**db:bridge-preflight**` → repeat save/quit if the lock returns → `**open**` **Unity Editor** on `**REPO_ROOT`**, wait up to **60s** for the lock → `**db:bridge-playmode-smoke`** (optional seed: `**npm run verify:local -- "x,y"**`). `**npm run verify:post-implementation**` is an **alias** for `**verify:local`**. **Non-macOS:** same through `**db:bridge-preflight`**, then manual **Editor** + smoke. **Not** for **CI**; requires **Postgres** and repo `**.env`** / `**config/postgres-dev.json**` as applicable, plus **Accessibility** for **System Events** when save/quit runs. `**unity:compile-check`** loads `**UNITY_EDITOR_PATH**` from `**.env**` inside the bash script (`[load-repo-env.inc.sh](../tools/scripts/load-repo-env.inc.sh)`) — **AI agents** must **not** skip `**npm run unity:compile-check`** because `**$UNITY_EDITOR_PATH**` is empty in the shell. Default behavior — no separate env toggle. See `[ARCHITECTURE.md](../ARCHITECTURE.md)` **Local verification**.

See also `[AGENTS.md](../AGENTS.md)` (Before You Start) and `[ia/skills/README.md](../ia/skills/README.md)`.

### Bridge export sugar tools

Thin MCP wrappers (`**unity_export_cell_chunk**`, `**unity_export_sorting_debug**`) reduce enqueue + poll boilerplate for the two parameterized **Editor** export kinds. They share the `**agent_bridge_job`** queue (`**[0008_agent_bridge_job.sql](../db/migrations/0008_agent_bridge_job.sql)**`), `**DATABASE_URL**` / `**config/postgres-dev.json**` requirements, and `**AgentBridgeCommandRunner**` dequeue behavior with `**unity_bridge_command**`. Dequeue / completion scripts live beside the migration in `**docs/postgres-ia-dev-setup.md**` (**IDE agent bridge**). Prefer `**unity_bridge_command`** when you need an arbitrary `**kind**`, `**unity_bridge_get**` with a `**command_id**` you already hold, or fine-grained `**timeout_ms**` on the raw command envelope.

### Composite context tools (pending registration — Stage 9 forward-looking)

> The tools below are **documented here for forward-compatibility** but are **not yet registered** in `tools/mcp-ia-server/src/index.ts`. Agent bodies that reference them must include a **Bash fallback** section listing the equivalent sequential calls. Registration lands in a future Stage 9+ task.

#### `issue_context_bundle`

One-shot context load for a single issue. Reads `ia/backlog/{id}.yaml` or `ia/backlog-archive/{id}.yaml`, owning spec `ia/projects/{id}.md`, parent master-plan stage block (via `parent_plan` + `task_key`), glossary anchors keyed on spec keywords, invariants slice, pair-contract rule slice. Replaces the 4–8 call opening sequence (`backlog_issue` + `router_for_task` + `glossary_discover` + `spec_section` + `invariants_summary` + `rule_content`) used in pre-Stage-9 skill / agent bodies.

**Input:** `{ issue_id: string }`.

**Output:** `{ issue: {...}, spec: {...}, master_plan_stage: {...}, glossary_anchors: [...], invariants: [...], pair_contract: {...} }`.

#### `lifecycle_stage_context`

Same shape for Stage-scoped bulk skills (`plan-author`, `stage-file-plan`, `stage-file-apply`, `stage-closeout-plan`, `plan-applier` Mode stage-closeout). Reads N Task specs + Stage header + shared MCP bundle in one call.

**Input:** `{ master_plan_path: string, stage_id: string }`.

**Output:** `{ stage: {...}, tasks: [{id, spec, ...}], glossary_anchors: [...], invariants: [...], pair_contract: {...} }`.

#### `orchestrator_snapshot`

Umbrella-level composite for `master-plan-new` / `master-plan-extend` / `release-rollout`. Reads orchestrator + rollout tracker + exploration doc + glossary anchors in one call.

**Input:** `{ slug: string }`.

**Output:** `{ orchestrator: {...}, rollout_tracker: {...}, exploration_doc: {...}, glossary_anchors: [...] }`.

### Computational tools vs spec slices

Use `**spec_section`**, `**spec_sections**`, `**glossary_***`, and `**router_for_task**` when you need **authoritative prose**, definitions, or routing from the Markdown IA corpus. Use **computational** tools (`isometric_world_to_grid`, `growth_ring_classify`, `grid_distance`, `pathfinding_cost_preview`, `geography_init_params_validate`) for **small deterministic numeric / validation** checks derived from the same rules as the game or interchange schemas. They **do not** replace specs: e.g. `**pathfinding_cost_preview`** v1 is explicitly **not** the full geo §10 cost model. Heavy grid queries (`**desirability_top_cells`**) stay `**NOT_AVAILABLE**` until Unity batchmode hooks land — charter trace `[BACKLOG-ARCHIVE.md](../BACKLOG-ARCHIVE.md)`.

## Server split architecture (TECH-524 / B1)

**Two-server shape.** `territory-ia` (IA-core) + `territory-ia-bridge` (Unity-bridge + compute) registered side-by-side in `.mcp.json`. IA-authoring sessions (design-explore, master-plan-new, stage-file-plan, author, audit, code-review) consume IA-core alone; verify-loop + spec-implementer + test-mode seams opt-in to bridge. Goal: shrink per-session tool manifest + token footprint for authoring-heavy loops.

`**MCP_SPLIT_SERVERS` flag.** Env var on the `territory-ia` server entry. Default `"0"` — single-server mode, `index.ts` registers both buckets (backward compat). Set `"1"` — standalone mode, IA-core only; bridge tools hidden from that server's `tools/list`. Flip timeline: default stays `0` through Stage 1.2; Stage 1.3 post-sweep (T1.3.6) flips default to `1` after per-theme attribution confirms correctness per NB-6 resolution (see `ia/projects/session-token-latency-master-plan.md`).

**Entry selection.** `tools/mcp-ia-server/bin/launch.mjs` reads `MCP_ENTRY`: `"index"` (default, backward-compat combined), `"index-ia"` (IA-core standalone), `"index-bridge"` (bridge + compute standalone). `.mcp.json` `territory-ia-bridge` entry sets `MCP_ENTRY=index-bridge`. Invalid values fall back to `"index"`.

**Tool buckets.** IA-core (≥22 tools): specs, rules, glossary, backlog, router, invariants, journal, reserve, materialize, plan-apply, csharp-class-summary, master-plan helpers. Bridge (14 registrations — `unity_bridge_command`, `unity_export_cell_chunk`, `unity_export_sorting_debug`, `unity_bridge_lease`, `unity_callers_of`, `unity_subscribers_of`, `findobjectoftype_scan`, `city_metrics_query`, plus 6 compute: `isometric_world_to_grid`, `growth_ring_classify`, `grid_distance`, `pathfinding_cost_preview`, `geography_init_params_validate`, `desirability_top_cells`). Shared register helpers: `tools/mcp-ia-server/src/server-registrations.ts`.
