# Web Platform — Exploration

**Status:** Interview in progress — not ready for design-expand  
**Scope:** Unified web presence for Territory Developer — DevOps dashboard + public game site + future user portal  
**Lifecycle stage:** Pre–`/design-explore` (exploration seed)

---

## Problem statement

Two distinct but related needs:

1. **Dev-facing dashboard** — `progress.html` is a static snapshot today; it needs to become a live, filterable DevOps view that reflects plan state without a manual `npm run progress` rebuild.
2. **Public-facing web presence** — the game needs a public site (landing, wiki, about, install, history) that shares infrastructure with the dev dashboard but exposes only curated surfaces.

A third surface looms: a **user portal** (save/map management, future payment) that requires auth, a real API, and persistent user data — well beyond the current static-HTML + MCP-only backend.

## Axes of exploration

### A — Progress dashboard improvements
- Live vs. static regeneration
- Filtering / drill-down (per-plan, per-step)
- GitHub Actions integration (auto-regen on push)
- Internal-only access or public read-only

### B — Public game site
- Landing / hero
- Wiki (images, mechanics, glossary)
- Project history / devlog
- About the developer
- Install guide
- Future: payment gateway (one-time purchase)

### C — User portal (future)
- Auth (accounts)
- Cloud saves / map management
- User-specific game actions via API
- Entitlement check post-payment

### D — Architecture
- Shared backend vs. separate services
- Hosting / deployment strategy
- Domain strategy
- Content management (static vs. CMS vs. MDX)
- Payment provider
- Auth provider
